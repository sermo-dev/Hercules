# Ticket 017: Strengthen AssumeUTXO Trust Model

## Summary

The current AssumeUTXO trust anchor in `assumeutxo.rs` is a SHA256 hash over Hercules's HUTX file format. It catches transit corruption and detects tampering of the file *given* the constants are honest, but it has no external verifiability — a careful Bitcoin user running their own Bitcoin Core full node cannot directly compare Hercules's commitment to anything Core publishes. The trust anchor is "trust the Hercules developers."

This ticket proposes three layered improvements that don't change Bitcoin's protocol-level trust assumptions (which can't be changed), but meaningfully strengthen the social-consensus and forward-verification properties of Hercules's snapshot:

1. **Portable UTXO hash format (MuHash)** — make the commitment hash reproducible from any Bitcoin Core full node, not just from Hercules's own conversion tool
2. **Multi-source attestations** — bake in commitments from multiple independent reviewers, require unanimous agreement
3. **Forward-validation confidence display** — surface the count of post-snapshot blocks successfully validated as a UI confidence indicator that grows over time

None of these provide protocol-level trustless verification, because Bitcoin doesn't have UTXO commitments in consensus and never has. They move Hercules from "trust one developer" to "trust the same developers Core users already trust, plus N independent reviewers, plus accumulating empirical evidence over time."

## Background

### What the current model commits to

`hercules-core/src/assumeutxo.rs` defines three constants:

1. `ASSUMEUTXO_HEIGHT = 943_500` — the block height at which the snapshot was taken
2. `ASSUMEUTXO_BLOCK_HASH` — the block hash at that height
3. `ASSUMEUTXO_HASH` — SHA256 of the HUTX-format file containing the UTXO set

The header comment says: "Generated by `convert_snapshot` from a Bitcoin Core `dumptxoutset` taken at block 943,500."

So the lineage is: Core full node → `dumptxoutset` → Hercules's `convert_snapshot` tool → HUTX file → SHA256 → `ASSUMEUTXO_HASH`.

### What's actually verified at load time

Per the existing `utxo.rs::load_snapshot` and the related code paths:

| Check | Catches |
|---|---|
| File SHA256 matches `ASSUMEUTXO_HASH` | Network corruption, hosting tampering, disk corruption — **given the constants are honest** |
| `ASSUMEUTXO_BLOCK_HASH` matches the chain at `ASSUMEUTXO_HEIGHT` | Wrong block hash compiled into the binary — verifiable trustlessly via PoW header chain |
| HUTX header internal consistency | Format-level corruption inside the file |

What's **not** checked:

| Gap | Consequence |
|---|---|
| `ASSUMEUTXO_HASH` against any external authority | A malicious Hercules build could ship a garbage UTXO set + matching garbage hash. The file SHA256 check would pass. The user would never notice. |
| Cross-implementation portability of the hash | Even a careful user running their own Core full node cannot directly compare Hercules's hash to Core's `dumptxoutset` hash — they're over different file formats |

### Why this matters

Bitcoin's whole value proposition is that you can verify the chain yourself rather than trusting third parties. AssumeUTXO is an explicit, well-understood compromise on that property — Bitcoin Core itself ships with hardcoded AssumeUTXO commits that users trust. The compromise is acceptable because:

- Multiple Core developers cross-check the commitments before merging
- The commits are public and auditable in the Core git history
- Users who don't accept the trade-off can disable AssumeUTXO and validate from genesis
- The commitment is reproducible: anyone running Core can verify

Hercules currently fails on the third and fourth points: there is no public attestation infrastructure beyond "the Hercules repo," and the commitment is **not directly reproducible** from a Core full node because of the format mismatch. A user who wants to verify Hercules's snapshot today has to:

1. Run Bitcoin Core to height 943,500
2. Run `dumptxoutset`
3. Run Hercules's `convert_snapshot` tool against the dump
4. SHA256 the resulting HUTX file
5. Compare to `ASSUMEUTXO_HASH`

Step 3 is itself trusting Hercules's code (a malicious `convert_snapshot` could produce different bytes from honest input). The verification path routes through Hercules's own tooling, defeating the point.

## Why the current design was chosen (and its real advantages)

Before proposing changes, it's worth being honest about why the file-hash approach was the natural first choice. It has genuine advantages:

1. **Single-pass streaming verification.** SHA256 over the file as it streams in from the network is trivially fast and memory-bounded — no need to parse the file format, walk the UTXO entries, or build any intermediate data structure. The verifier is `for chunk in stream { hasher.update(chunk) }`. On iOS this is hardware-accelerated via CommonCrypto / Apple Silicon SHA-NI.

2. **Catches all byte-level tampering, not just UTXO-set differences.** A file hash detects any modification anywhere in the file — header, body, padding, ordering. Format-aware hashes (MuHash, sorted-tuples SHA256) only commit to the *semantic* UTXO set, so a tampered HUTX file with corrupted header bytes but correct UTXO content would pass a semantic check but fail a file check. Both kinds of detection are valuable.

3. **No format dependency in the verifier.** The file hash doesn't need to know what's inside the file. It works for any binary blob. This made HUTX iteration cheap during early development — change the file format, regenerate the hash, ship.

4. **Conversion-time hashing.** The hash is computed once at snapshot generation time on the build server. Verification at runtime is just SHA256, no UTXO library involved. This kept `hercules-core` lean and avoided pulling in MuHash code or canonicalization logic.

5. **iOS performance under battery constraints.** Streaming SHA256 of an 8 GB file on iPhone Pro hardware is on the order of 30-60 seconds and consumes minimal CPU. MuHash over the same data requires parsing the format, decoding ~80M UTXO entries, and updating the accumulator per entry — significantly more work, more memory pressure, and more battery cost during the snapshot import path that's already long.

6. **Defense against hosting compromise.** The current model fully prevents Cloudflare R2 from substituting a malicious file — any byte change is caught. The file hash *is* the right tool for this specific threat, and this protection should remain after this ticket lands.

The current design isn't wrong — it's the obvious and reasonable starting point for a single-trust-anchor model. The gap this ticket addresses is **external verifiability**, not the on-device verification path. Both should coexist.

## Design

Three layers, each independently valuable, ship in order.

### Layer 1 — Portable UTXO hash (MuHash)

**The change**: in addition to the existing file SHA256, compute and ship a **content-based** UTXO hash that's reproducible from any Bitcoin Core full node, independent of file format.

**Recommended algorithm**: MuHash. Reasons:
- Bitcoin Core uses MuHash internally as its UTXO set hash (since #19055, it's the algorithm behind `gettxoutsetinfo`'s `muhash` field)
- A Core user can produce the MuHash of the UTXO set at any height with a single command (`bitcoin-cli gettxoutsetinfo muhash <height>`)
- MuHash is set-independent: the order in which UTXOs are inserted doesn't affect the result, so canonicalization of entry order is not required
- Library: `secp256k1`-based, already in the Bitcoin/Rust ecosystem (`rust-bitcoin` exposes it)

**Alternative considered**: canonical sorted SHA256 over `(txid, vout, value, scriptPubKey)` tuples. Simpler to implement (no MuHash library), but requires defining a strict canonical ordering and serialization, and doesn't match what Core's CLI exposes natively. Reject in favor of MuHash for the matching-Core-output property.

**New constants in `assumeutxo.rs`**:

```rust
/// MuHash of the UTXO set at ASSUMEUTXO_HEIGHT, matching the value
/// produced by `bitcoin-cli gettxoutsetinfo muhash 943500` on a fully
/// validating Bitcoin Core node. This commitment is portable across
/// implementations and can be verified by any Core user.
pub const ASSUMEUTXO_MUHASH: [u8; 32] = [...];
```

**New verification path in `utxo.rs::load_snapshot`**:

1. (Existing) Verify file SHA256 against `ASSUMEUTXO_HASH` — catches transit/hosting tampering
2. (New) Stream the file's UTXO entries through a MuHash accumulator
3. (New) Compare the resulting MuHash against `ASSUMEUTXO_MUHASH`
4. Both must pass before commit

This adds runtime cost: MuHash over 80M UTXOs is meaningful work (~1-2 minutes on iPhone Pro hardware, rough estimate, needs measurement). Mitigation: do it concurrently with the file SHA256 pass and the database insertion pass — they all walk the file once, so a careful streaming pipeline only reads bytes from disk once total. Wall-clock cost is dominated by I/O, not by either hash.

**External verification flow (the win)**:

1. User runs their own Bitcoin Core to height 943,500
2. User runs `bitcoin-cli gettxoutsetinfo muhash 943500`
3. User compares the output to `ASSUMEUTXO_MUHASH` in the Hercules source

Two commands, no Hercules tooling involved. The trust check now routes around Hercules entirely.

### Layer 2 — Multi-source attestations

**The change**: instead of a single MuHash baked into the binary, ship **N hashes from independent reviewers** who each ran their own Core full node and produced the same MuHash. Require unanimous agreement at startup.

**File format**: `attestations.toml` (or similar) embedded into the binary at build time:

```toml
[[attestation]]
reviewer = "Hercules build server"
key_fingerprint = "..."
muhash = "..."
signed_at = "2026-02-01T00:00:00Z"
signature = "..."

[[attestation]]
reviewer = "Reviewer A"
key_fingerprint = "..."
muhash = "..."
signed_at = "2026-02-03T00:00:00Z"
signature = "..."

# ... at least 3 attestations total
```

Each attestation is a signature over `(height, block_hash, muhash)` by a key the user trusts (or that the Hercules project documents and rotates publicly). At load time, Hercules verifies all signatures, checks that all `muhash` values match, and refuses to load if any disagree.

**The win**: trust shifts from "Hercules developers" to "any one of N reviewers being honest." If the Hercules build server is compromised but Reviewer A's key is not, the compromise is caught immediately at load — the muhash values won't match and the load aborts with a loud error.

**Operational requirement**: Hercules needs at least 2-3 named reviewers committed to running Core and signing each new snapshot. This is a relationship/governance question, not a code question, but it's the gating constraint for Layer 2.

### Layer 3 — Forward-validation confidence display

**The change**: surface the cumulative count of post-snapshot blocks Hercules has successfully validated against the loaded UTXO set, as a tangible confidence indicator that grows over time.

**Why this matters**: every block validated after the snapshot is implicit cross-check. If the loaded UTXO set were wrong (extra fake coins, missing real coins, wrong scriptPubKeys), then sooner or later a real block would touch a UTXO Hercules has wrong, and validation would fail. The longer Hercules runs without a validation failure, the higher the confidence that the snapshot was honest.

**Caveats** (must be communicated honestly in the UI):

- Only catches differences in UTXOs that get *touched* by post-snapshot blocks. Untouched garbage UTXOs are invisible forever.
- Detects "added" UTXOs (their spends fail script validation) but is mostly blind to "removed" UTXOs (legitimate UTXOs Hercules deleted just look unspendable to the user — only noticed if they try to spend those specific coins).
- Not a substitute for the hash verification at load. Defense in depth, not a replacement.

**UI design**:

- Settings → "About / Trust" card shows:
  - "Snapshot loaded at block 943,500"
  - "Verified by N attestations" (from Layer 2)
  - "Forward-validated against 1,247 subsequent blocks"
  - Tooltip: "Each block validated after the snapshot is an implicit consistency check. Higher numbers indicate stronger empirical confidence, but are not a substitute for the hash verification at snapshot load."
- Counter increments on each successful block validation, persists to disk so it survives restarts
- Resets only when the snapshot itself is reloaded (e.g., the user wipes state and restarts in Validate-from-Genesis mode and back)

This is pure UI work over data Hercules already has — the validation count is implicit in the chain tip minus the snapshot height. ~50 lines.

## Phasing

| Layer | Effort | Independently shippable? |
|---|---|---|
| 1 — Portable MuHash | ~300 lines Rust + measurement work | **Yes** — adds verification, doesn't replace anything |
| 2 — Multi-source attestations | ~200 lines Rust + governance work | Yes — depends on Layer 1 conceptually but doesn't require it |
| 3 — Forward-validation counter | ~50 lines Swift + ~30 lines Rust | **Yes** — pure additive UI, no dependencies |

Recommended order: **3 → 1 → 2**. Layer 3 ships immediately for free. Layer 1 is the highest-impact technical change (enables external verification). Layer 2 captures the social-trust improvement once the governance is in place.

## Verification

**Layer 1**:
- Run `bitcoin-cli gettxoutsetinfo muhash 943500` against a fresh Core sync and compare against `ASSUMEUTXO_MUHASH` — must match exactly
- Tamper with one UTXO entry in the HUTX file, observe MuHash check fails (file SHA256 also fails — both layers catch it)
- Tamper with `ASSUMEUTXO_MUHASH` constant, rebuild, run snapshot load, observe load aborts with clear error before any database commit
- Measure on-device: file SHA256 + MuHash accumulator + DB insertion pipelined into a single file read pass — wall-clock cost should be within 20% of the existing single-pass

**Layer 2**:
- Tamper with one attestation's `muhash` field, observe load aborts at signature verification or muhash agreement check
- Tamper with one attestation's signature, observe load aborts at signature verification
- Remove an attestation, observe load aborts with "insufficient attestations"
- Add a new valid attestation, observe load proceeds

**Layer 3**:
- Load snapshot fresh, observe counter starts at 0
- Validate 10 blocks, observe counter advances
- Restart app, observe counter persists and continues advancing
- Reset to Genesis mode, observe counter resets

## Dependencies

- `secp256k1` MuHash bindings (`rust-bitcoin` exposes these via `bitcoin::network::AddrV2::*` family — check current API surface)
- Build pipeline change: snapshot generation must compute MuHash and emit it for inclusion in `assumeutxo.rs`
- Governance: identifying and onboarding 2-3 named reviewers willing to attest each snapshot (Layer 2 only)
- Coordination with **ticket 012** (Tor-routed snapshot download) — both touch the snapshot trust path and should land in a coherent order

## Estimated effort

- **Layer 1**: ~300 lines Rust (MuHash accumulator integration into the streaming load path), ~1 day to instrument and measure on-device performance, build pipeline update
- **Layer 2**: ~200 lines Rust (signature verification, attestation parsing, agreement check), substantial **non-code** work to establish the reviewer relationships and signing flow
- **Layer 3**: ~80 lines total across Rust counter persistence and Swift UI

Roughly a week of focused engineering for Layers 1 and 3. Layer 2 is gated on the governance side, not the code side.

## Open questions

- **MuHash availability in `rust-bitcoin`**. The crate exposes MuHash via `bitcoin::p2p::*` and related modules in recent versions, but the API surface has changed across releases. Need to confirm what's available in the version Hercules pins. If unavailable, vendor `bitcoin-muhash` from a separate crate or hand-implement (~150 lines, references the BIP).
- **Wall-clock cost of MuHash on-device**. The estimate of 1-2 minutes is rough. If actual measurement shows >5 minutes added to snapshot import, may need to gate the full MuHash check behind an opt-in "extra paranoid" toggle and ship Layer 1 with a sampled check (e.g., MuHash over a deterministic 10% sample) as the default. Measure first.
- **Number of attestations required**. 3 is the proposed minimum for Layer 2 — gives meaningful Byzantine tolerance (any 1 dissenter still leaves 2-vs-1 disagreement → caught). 5 is stronger but requires more reviewer commitment. 2 is a single point of failure (a 1-1 disagreement is ambiguous). Recommend 3, document the trade-off.
- **Reviewer key rotation**. If a reviewer's key is compromised or they step down, Hercules needs to ship a new release with the updated attestation set. This is fine for releases, but in-the-wild old binaries continue to trust the old key. Standard supply-chain problem; standard answer is "release frequently and document key rotation publicly."
- **Should Layer 3 also track how many *spends* of original snapshot UTXOs have succeeded?** A tighter metric than "blocks validated" — directly counts the number of coins from the snapshot that have been successfully spent without script-validation failure. More expensive to track but more semantically meaningful as a confidence indicator.
- **Genesis-mode escape hatch**. The existing `SettingsView::validationModeCard` toggle to switch to "Validate from Genesis" mode should remain regardless of this ticket — it's the ultimate escape hatch for users who reject any AssumeUTXO trust assumption, and it's the only fully-trustless option Bitcoin protocol allows.
